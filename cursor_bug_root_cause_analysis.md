# Cursor Bug Root Cause Analysis - Deep Dive

## Executive Summary
When a user refreshes their tab, other users can no longer see their cursor. 
There are **TWO separate bugs** causing the cursor visibility issue after refresh:

1. **Sequence Number Bug**: When a user refreshes, their sequence counter resets to 0, but other clients reject all updates as "stale" because they still have the old high sequence number
2. **Cleanup Bug**: When a peer disconnects, the removal handler fails to clear their smoother because it looks up by clientID but the map is keyed by userId
  - **Yjs clientID**: Auto-assigned by Yjs, changes every time a client connects (including refreshes)
  - **Stable userId**: Generated by UserProfileManager, persists across refreshes via localStorage
  - `yAwareness.getStates()` returns a Map<**clientId**, state>
  - `peerSmoothers` is a Map<**userId**, PeerSmoothing>
  - `cursorTrails` in presence-cursors.ts is a Map<**userId**, CursorTrail>

## Bug #1: The Sequence Number Reset Problem

### What Happens
When Tab A refreshes:
1. Tab A's `awarenessSeq` resets to 0 (new RoomDocManager instance)
2. Tab B and C have `peerSmoothers.get("user-abc").lastSeq = 100` (from before refresh)
3. Tab A sends awareness with `seq: 1`
4. Tab B and C's `ingestAwareness()` drops it: `if (seq <= ps.lastSeq) return;` (1 <= 100)
5. **Result**: No cursor updates are ever accepted from the refreshed client

### The Code Evidence
```typescript
// room-doc-manager.ts line 483
if (typeof seq === 'number' && seq <= ps.lastSeq) return;
```

This line ALWAYS rejects updates after a refresh because the sequence number can never catch up to the old value.

## Bug #2: The Key Mismatch Problem
In `/client/src/lib/room-doc-manager.ts`,
### What Happens
When a peer disconnects:
```typescript
// Line 1766-1774 - The broken cleanup
else if (!state && clientId) {
  // Handle removed peers
  const ps = this.peerSmoothers.get(String(clientId)); // BUG: Using clientId instead of userId!
  if (ps) {
    ps.hasCursor = false;
    ps.prev = ps.last = ps.displayStart = undefined;
    ps.animStartMs = ps.animEndMs = undefined;
  }
}
```

- `peerSmoothers` is a `Map<userId, PeerSmoothing>`
- But we're trying to look up by `clientId`
- This lookup ALWAYS fails
- The old smoother with stale `lastSeq` is never cleared

## Why Both Bugs Matter

Even if we fixed the cleanup bug, the sequence number bug would still break cursors. Even if we fixed the sequence bug, we'd accumulate stale smoothers over time.

## The Optimal Solution

After analyzing the architecture, the cleanest fix is to **key smoothers by clientID instead of userId**:

### Also: Self filtering should be tab based: Approach: Filter by ClientID
```typescript
// NEW: Check clientId !== this.yAwareness.clientID instead of userId check
if (state.userId && clientId !== this.yAwareness.clientID) {
  // Process this peer's awareness
}
```
### Why This Is Better

1. **Natural alignment with Yjs**: Awareness states are keyed by clientID
2. **Automatic cleanup**: When clientID is removed, we can clean up correctly
3. **Fresh state on reconnect**: New connection = new clientID = new smoother = seq starts at -1
4. **No mapping needed**: No need for clientID→userId lookup tables
5. **Simpler code**: Removal logic "just works"
6.  **Semantic Correctness**
- `clientID` represents a unique connection/session
- Filtering by `clientID` means "don't process my own connection's awareness"
- This is the actual intent - we want to filter out THIS browser tab, not ALL tabs from this user
7. **Multi-Tab Behavior**
With your approach when same user has Tab A and Tab B open:
- Tab A sees Tab B's cursor ✓ (different clientIDs)
- Tab B sees Tab A's cursor ✓ (different clientIDs)
- Other users see ONE cursor (last update wins in userId-keyed Map)
- UserAvatarCluster shows ONE user (deduped by userId)
This is actually GOOD because:
- Users can debug their own multi-tab behavior
- No special casing needed
- Natural and predictable
**Refresh Scenario - Perfect Fix**
When Tab A refreshes:
1. Old clientID 1 removed → `peerSmoothers.delete(1)` ✓ Works!
2. New clientID 2 added → `peerSmoothers.set(2, {lastSeq: -1})` ✓ Fresh state!
3. Seq 0 > -1, updates accepted immediately ✓ No stale seq issue!

**The filtering works perfectly**. Each tab filters out its own cursor (by clientID) but shows all other cursors, even from the same userId. The presence-cursors.ts layer is a pure renderer - it draws whatever cursors are in the Map without any additional filtering. 
The `users` Map is still keyed by userId, so:
- UserAvatarCluster sees one user per userId ✓
- User count is correct ✓
- "ME" identification still works ✓

- Thus: we fix the bug, and additionally have good UX for displaying one user in two tab (same profile) scenarios, while still rendering the cursor in the other tab

## Why This Fixes Both Bugs

### Bug 1: Sequence Number Reset
- New connection = new clientID = new smoother with lastSeq: -1
- Seq 0 from refreshed client is > -1, so updates are accepted

### Bug 2: Cleanup Failure
- Smoothers keyed by clientID matches the removal event's clientID
- Cleanup lookup now succeeds and removes stale data

## Why NOT Use Other Approaches

### Approach: Clearing Remote Peer Awareness States on Disconnect
- **Hard no**: Individual awareness: Each client is responsible for managing its own awareness state. A client disconnecting should not have the power to clear the states of other clients.

### Approach: ClientID→UserID Mapping
- **Cons**: Extra data structure to maintain, complex cleanup, doesn't fix the core mismatch
- **Why it's worse**: We'd still be fighting the natural Yjs architecture

### Approach: Reset Seq on Reconnect Detection
- **Cons**: How do we detect reconnect? Would need session IDs or complex heuristics
- **Why it's worse**: Doesn't fix the cleanup bug, adds complexity

### Approach: Ignore Sequence Numbers
- **Cons**: Loses the benefit of dropping out-of-order packets
- **Why it's worse**: Could cause cursor jitter with network reordering

## Conclusion

The bug is actually two bugs compounding each other. The cleanest fix is to align our data structures with Yjs's natural model by keying smoothers by clientID. This solves both bugs elegantly without additional complexity. 
The fix aligns our data structures with Yjs's natural clientID-based model. By filtering awareness by clientID (not userId) and keying smoothers by clientID, we get:
- Natural cleanup on disconnect
- Fresh state on reconnect
- Multi-tab cursor visibility (useful!)
- Same UI behavior (one avatar per unique user)
